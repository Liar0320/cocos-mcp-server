# Cocos Creator MCP Server PRD

> 文档性质：基于当前仓库实现反向整理的 PRD 基线  
> 基线时间：2026-04-08  
> 主要依据：`source/main.ts`、`source/mcp-server.ts`、`source/tools/*.ts`、`source/panels/default/index.ts`

## 1. 产品概述

`cocos-mcp-server` 是一个运行在 Cocos Creator 3.8.6+ 内部的扩展插件。它在编辑器内提供可视化面板用于服务器启停和工具开关管理，并在本地启动一个 HTTP MCP 服务，使 AI 助手或外部自动化客户端能够通过标准 MCP 调用和简化 HTTP API 调用方式控制 Cocos Creator 编辑器。

产品的核心价值不是“再造一个编辑器界面”，而是把 Cocos Creator 的常用编辑能力、场景/节点/组件/预制体/资源操作和调试查询能力标准化为可被 AI 调用的工具层。

## 2. 背景与问题

### 2.1 现有问题

- AI 助手虽然能生成 Cocos 代码，但缺少对编辑器运行态、场景树、资源库和预制体系统的直接操作能力。
- Cocos Creator 的很多编辑器能力分散在 `Editor.Message`、场景脚本、资源数据库、构建面板等不同接口中，AI 很难稳定编排。
- 用户接入 MCP 时通常还要自己处理服务启动、端口配置、工具列表、协议格式和 JSON 转义问题。

### 2.2 产品机会

- 通过一个可在编辑器内直接安装的扩展，把 AI 接入成本降到“打开面板 -> 启动服务 -> 配置客户端”。
- 把高频编辑器操作封装成稳定的工具定义和统一输入 Schema，降低 AI 调用失败率。
- 提供工具启用/禁用配置，允许用户按项目或任务控制暴露给 AI 的能力范围。

## 3. 产品目标

### 3.1 业务目标

- 让 AI 助手可以在 Cocos Creator 内完成大部分高频编辑器操作。
- 降低用户把 Cocos 项目接入 MCP 的配置成本。
- 提供一套可维护、可筛选、可扩展的工具注册机制。

### 3.2 产品目标

- 在编辑器内提供服务器管理面板。
- 通过 HTTP 暴露标准 MCP `initialize`、`tools/list`、`tools/call`。
- 提供简化 REST 风格 API 以便脚本或调试使用。
- 覆盖场景、节点、组件、预制体、资源、构建、调试、场景视图、参考图和偏好设置等能力域。
- 支持工具启用状态持久化。

### 3.3 非目标

- 不作为独立 Web 服务运行，必须依附 Cocos Creator 扩展宿主。
- 不提供远程公网服务，当前实现只绑定 `127.0.0.1`。
- 不负责 AI 模型本身的对话、规划或提示词工程。
- 不提供成熟自动化测试框架；当前仓库仍以手工验证为主。

## 4. 目标用户

### 4.1 直接用户

- 使用 Claude CLI、Cursor 或其他 MCP 客户端的 Cocos Creator 开发者。
- 希望用 AI 在编辑器内执行结构化操作的技术美术、关卡设计或工具开发者。

### 4.2 间接用户

- 需要为团队提供标准化 AI 编辑器接入方案的技术负责人。
- 需要在项目中进行可重复资源处理、场景巡检或构建辅助的脚本作者。

## 5. 核心使用场景

### 5.1 场景编辑

- AI 打开场景、读取层级、查询节点、创建节点、挂组件、修改属性、保存场景。

### 5.2 预制体生产与修复

- AI 从节点创建预制体、实例化预制体、同步或还原实例、校验预制体格式。

### 5.3 资源管理

- AI 查询资源、导入/复制/移动/删除资源、查看依赖、导出资源清单、批量处理贴图。

### 5.4 编辑器调试与诊断

- AI 读取控制台/项目日志、获取编辑器信息、执行调试脚本、检查场景问题。

### 5.5 视觉辅助

- AI 在 Scene View 中控制 Gizmo、网格、2D/3D 视图、聚焦镜头，或加载参考图进行对齐。

### 5.6 接入与治理

- 用户在扩展面板里启动服务、设置端口、复制 URL，并按需启用或禁用工具。

## 6. 关键用户流程

### 6.1 首次接入

1. 在 Cocos Creator 中安装并加载扩展。
2. 打开 MCP 面板。
3. 设置端口、自动启动、调试日志等参数。
4. 启动本地 HTTP 服务。
5. 在外部 MCP 客户端中配置 `http://127.0.0.1:{port}/mcp`。

### 6.2 日常使用

1. AI 调用 `tools/list` 获取可用工具。
2. AI 基于工具名和 Schema 发起 `tools/call`。
3. 工具内部通过 `Editor.Message`、资源数据库、场景脚本等 Cocos API 执行操作。
4. 结果经 MCP 文本内容返回给客户端。

### 6.3 工具治理

1. 用户进入“工具管理”页签。
2. 查看按分类聚合的工具清单。
3. 按单个工具或整个分类启用/禁用。
4. 保存更改，后端重新过滤对外暴露的工具。

## 7. 功能范围

### 7.1 扩展面板

- 查看服务运行状态。
- 启停服务。
- 编辑端口、自动启动、调试日志、最大连接数配置。
- 复制本地 HTTP 地址。
- 查看工具总数/启用数/禁用数。
- 批量全选/取消全选工具。

### 7.2 MCP 服务层

- `initialize`
- `tools/list`
- `tools/call`
- `GET /health`
- `GET /api/tools`
- `POST /api/{category}/{tool}`

## 8. 功能需求

### FR-1 服务器管理

- 系统必须允许用户在编辑器面板中启动和停止 MCP 服务。
- 系统必须支持保存端口和自动启动设置到项目 `settings/` 目录。
- 系统必须在服务运行时显示本地可访问 URL。
- 系统必须默认绑定 `127.0.0.1`。

### FR-2 工具注册与过滤

- 系统必须从各工具类的 `getTools()` 动态收集工具定义。
- 系统必须支持按照启用配置过滤对外暴露的工具。
- 系统必须把启用状态持久化到项目 `settings/tool-manager.json`。

### FR-3 MCP 协议支持

- 系统必须支持 MCP 初始化和工具列表、工具调用三个基础方法。
- 系统返回结果必须兼容 MCP 文本内容封装格式。
- 系统应尽量容忍常见 JSON 转义错误。

### FR-4 编辑器控制

- 系统必须覆盖场景、节点、组件、预制体和资源的核心编辑能力。
- 系统应支持查询类工具和执行类工具共存。
- 系统应对关键高频操作提供清晰的输入 Schema。

### FR-5 调试与辅助

- 系统必须支持读取控制台/项目日志和编辑器环境信息。
- 系统应提供请求格式化与 JSON 修复能力，帮助 AI 降低调用失败率。
- 系统应支持 Scene View 与参考图辅助操作。

## 9. 非功能需求

### 9.1 兼容性

- 目标宿主为 Cocos Creator `>= 3.8.6`。
- 插件需在编辑器加载流程内完成初始化。

### 9.2 安全性

- 默认仅监听本机回环地址。
- 不应无条件对外暴露到局域网或公网。
- 工具治理应允许用户缩减暴露面。

### 9.3 可维护性

- 新工具应通过分类工具类扩展。
- 工具定义应包含名称、描述和输入 Schema。
- 文档应尽量与源码注册结果保持一致。

### 9.4 可观测性

- 服务启动、停止、工具数量、异常应输出日志。
- 面板应轮询展示当前服务状态。

## 10. 当前实现状态判断

### 10.1 已实现

- 编辑器内服务器面板与工具管理页签。
- HTTP MCP 服务与简化 API。
- 157 个工具的分域注册。
- 设置和工具启用状态写入项目 `settings/`。
- 基础 JSON 修复逻辑。

### 10.2 已暴露但实现不完整或存在偏差

- `allowedOrigins` 设置已定义，但服务端当前固定返回 `Access-Control-Allow-Origin: *`。
- `maxConnections` 设置已暴露在 UI 和配置类型中，但服务端当前没有连接上限控制逻辑。
- `clients` 状态字段存在，但 HTTP 模式下始终返回 `0`。
- `broadcast` 当前以模拟监听为主，`Editor.Message.on/off` 实际接入被注释。
- 仓库现有 `README.md`、`FEATURE_GUIDE_CN.md` 与测试脚本中的能力描述存在版本漂移。
- `source/test/mcp-tool-tester.ts` 仍以 WebSocket 方式示例调用，而当前服务实现为 HTTP MCP。

## 11. 成功标准

从当前产品视角，以下结果可视为成功：

- 用户可以在 Cocos Creator 中一键启动本地 MCP 服务。
- MCP 客户端能够成功完成 `initialize -> tools/list -> tools/call`。
- AI 可以在真实项目里完成常见场景、节点、组件、预制体和资源操作。
- 用户可以通过工具管理降低暴露给 AI 的工具集合。

## 12. 风险与待确认项

- 文档口径风险：现有 README/功能指南中的“50 工具”与“158 工具”均与当前源码 157 工具不一致。
- 协议感知风险：测试辅助文件仍保留旧的 WebSocket 使用方式，可能误导接入方。
- 广播能力风险：广播工具看起来已提供接口，但底层真正订阅 Editor 消息的实现尚未启用。
- 配置预期风险：UI 暴露了 `maxConnections` 和 `allowedOrigins`，但用户可能误以为这些设置已被严格执行。
- 返回格式风险：MCP `tools/call` 的业务结果被包装为 `content[0].text` 的 JSON 字符串，客户端通常需要二次解析。

## 13. 后续建议

- 把 README、功能指南、测试脚本和源码注册清单统一到同一版本口径。
- 明确哪些设置当前只是“配置占位”，哪些已真正生效。
- 为高风险工具域补充最小自动化回归用例。
- 给文档生成建立脚本化流程，避免以后再出现工具数量漂移。
